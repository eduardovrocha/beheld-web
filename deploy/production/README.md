# Production deploy

This dir is the deploy contract for the production VPS. It expects:

- `/opt/devprofile/web/` checked out from `eduardovrocha/devprofile-web`
  via a GitHub deploy key (read-only, generated on the VPS at
  `~/.ssh/github_devprofile_web`).
- Postgres 14 + Redis 6 running natively on the host, both bound to
  `127.0.0.1` only. UFW blocks 5432 and 6379 externally.
- Docker engine + Compose v2 plugin installed; `ubuntu` user in the
  `docker` group.
- Secrets in `/etc/devprofile/` (root-owned, perms 600):
  - `postgres.env` — generated by the VPS bootstrap; do not edit.
  - `app.env` — copy from `.env.example`, fill in the blanks.

## Files

| File | Purpose |
|---|---|
| `docker-compose.yml` | backend (Rails) + caddy (reverse proxy + static SPA) |
| `Caddyfile`          | routes `/api`, `/up`, `/bundles`, `/v/*` to Rails; everything else to the SPA |
| `.env.example`       | template for `/etc/devprofile/app.env` |
| `deploy.sh`          | one-shot: pull, build frontend, build backend, migrate, up |

## First deploy (operator runs once)

```sh
# 1. Populate secrets
sudo cp /opt/devprofile/web/deploy/production/.env.example /etc/devprofile/app.env
sudo chmod 600 /etc/devprofile/app.env
sudo nano /etc/devprofile/app.env
# Fill in: SECRET_KEY_BASE, DEVPROFILE_PLATFORM_PRIVATE_KEY,
#          GITHUB_OAUTH_CLIENT_SECRET.

# 2. Run the deploy
cd /opt/devprofile/web/deploy/production
bash deploy.sh
```

Subsequent deploys: same `bash deploy.sh` after pushing changes to GitHub.

## Generating SECRET_KEY_BASE

Locally:

```sh
cd web/source/backend
docker run --rm -v "$PWD:/app" -w /app ruby:3.3-slim \
  bash -c "gem install bundler && bundle install --without development test && bundle exec rails secret"
```

Or on the VPS once the backend image is built:

```sh
docker compose -f /opt/devprofile/web/deploy/production/docker-compose.yml \
  run --rm backend bin/rails secret
```

## Routing

| Path pattern              | Backend |
|---------------------------|---------|
| `/api/*` `/up` `/bundles` | Rails (always) |
| `/v/:id/summary`          | Rails |
| `/v/:id/badge.svg`        | Rails |
| `/v/:id` (Accept: JSON)   | Rails |
| `/v/:id` (Accept: HTML)   | SPA (React Router renders) |
| any other path            | SPA (with `try_files {path} /index.html` fallback) |

This gives the SPA full control over `/v/:id` rendering (so the IdentityTag
shows) while keeping Rails responsible for the JSON, summary, and badge
endpoints embedded badges from third-party sites need.

## Networking

`network_mode: host` for both containers — the Rails container reaches
host's Postgres / Redis on `127.0.0.1` directly, and Caddy binds 80/443
on the host. No port mapping, no Docker bridge in between.

This is the pragmatic choice for a single-VPS deploy. Move to bridged
networks + a docker-internal Postgres only when scaling to multi-node.

## TLS (post-DNS)

Once `devprofile.info` resolves to this VPS, edit `Caddyfile`:

```diff
- :80 {
-   ...
- }
+ devprofile.info {
+   ...
+ }
```

Remove the `auto_https off` block at the top. Caddy will provision a
Let's Encrypt cert on the first hit to port 443. Update the GitHub
OAuth App's callback URL to `https://devprofile.info/...`.
